## repo에서 확인된 사실 (질문의 근거 — 검증 소스)

- NestJS + WebSocket / Next.js Server Actions / Redis + PostgreSQL + Prisma 7.4 / JWT / Docker. TypeScript 98.9%.
- README에 **Write-back 캐싱 패턴으로 기존 CRUD 대비 DB I/O 90% 이상 감소** 명시 — 수치 존재!
- Lazy Update 전략: 블라인드 레벨을 타이머 대신 핸드 시작 시점 계산 → stateless 서버.
- N+1 쿼리를 컨텍스트 주입 + 배치 연산으로 해결 (README 트러블슈팅).
- 비동기 콜백 최적화 — 즉각 UX 피드백 + 데이터 정합성 유지 (README 트러블슈팅).
- 딜러 OTP 인증, 탈락/리바이 로직, 30초 자동 폴드/체크, 사이드팟 계산, 쇼다운 랭킹.
- 아키텍처 다이어그램·플로우차트 문서화 잘 됨. 미비: 배포 가이드, API 문서, 테스트.

## 코드 검증 결과 — "DB I/O 90% 이상 절감" 수치 (2026-07-03, 코드 직접 확인)

**구조적 근거는 실재. 단, 측정치가 아니라 추정치.**

확인한 것 (backend/src):
- 핸드 중 플레이어 액션 경로(`playsync.service.ts` `handleAction`): Redis `getSnapShot` → `TableEngine`(순수 로직, I/O 없음) → Redis `saveSnapShot`. **prisma 호출 0회** — `ws.gateway.ts`, `game-engine/table-engine.ts` 모두 DB 접근 없음.
- prisma 호출은 정산·경계 이벤트에만: `resolveWinners`, 탈락/리바이(`$transaction` + `updateMany`/`deleteMany`), `tournamentFinished`, 딜러 로그인, auth/user/store.
- **벤치마크·측정 코드는 저장소에 없음.** README의 90%는 로그/측정 기반이 아니라 구조에서 나온 추정.

판단:
- 핸드당 상태 변경 15~30회(6인 테이블, 베팅 4라운드 기준)가 전부 Redis, DB 트랜잭션은 핸드 종료 시 1~2회 → 산술적으로 87~97% 절감. **"90% 이상"은 구조적으로 타당한 범위.**
- 그러나 "기존 방식 대비"의 기준(모든 액션을 DB에 쓰는 가상의 CRUD 구현)이 실측 비교가 아니므로, 면접에서 "어떻게 측정했나"를 받으면 방어가 약함.
- 포트폴리오 표기 권장: ① 실측 확보(액션당 쿼리 카운트 로깅으로 실제 비율 산출) 또는 ② 사실 서술로 전환 — "핸드 중 상태 변경은 전부 Redis에서 처리하고 DB 트랜잭션은 정산 시점 1회만 실행" + 괄호로 산출 근거(핸드당 상태 변경 N회 → DB 쓰기 1회). ②가 즉시 가능하고 면접 방어 가능.
- → 아래 D질문 첫 항목에서 사용자와 표기 방식 결정.
